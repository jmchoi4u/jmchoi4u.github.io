블로그 개발자 사용 설명서

이 문서는 Jekyll + Chirpy + GitHub Pages 구조를 기준으로 작성한 상세 설명서입니다. 실제 동작 방식, 수정 포인트, 배포 흐름, SEO, 템플릿 관리 위치를 개발자 관점에서 정리했습니다.

빠른 이동: 구조 / 빌드 / 배포 / SEO / 웹 분석 / 댓글

1. 현재 구조 요약

경로 역할
_config.yml 사이트 전체 설정. 제목, 설명, URL, 소셜 링크, 프로필 이미지 경로, Jekyll exclude 설정
_data/contact.yml 사이드바 하단 연락처 아이콘 목록
_posts 게시글 원본 마크다운
_tabs About, Archives, Categories, Tags 등 상단/사이드 탭 페이지
assets/img 프로필 이미지와 글 이미지 보관
src/templates 게시글/페이지 템플릿 및 마크다운 예제 보관. Jekyll 출력 제외
app 블로그 작성 도우미 웹앱. 글 작성, 이미지 저장, 미리보기, 빌드 테스트, git push 담당
.github/workflows/pages-deploy.yml GitHub Pages 배포 워크플로

2. 프로필 이미지 동작 방식

Chirpy는 _config.yml의 avatar: 값을 읽어 사이드바 프로필 이미지를 렌더링합니다.

avatar: /assets/img/profile/avatar.png

실제 렌더링 시 원형 썸네일로 작게 보이므로, 아래 조건을 만족하는 파일이 가장 안정적입니다.

  • 정사각형 PNG 또는 JPG
  • 얼굴 중심 크롭
  • 권장 크기 512px 이상
  • ASCII 파일 이름 사용
현재 로컬 빌드 기준으로 avatar.png는 이미 정상 렌더링됩니다. 적용이 안 된 것이 아니라, 전신형 그림이 112px 원형으로 축소되어 체감이 약한 상태로 보는 편이 정확합니다.

3. 연락처와 소셜 링크

_config.yml

  • social.name: 사이트 작성자 이름, 푸터 저작권명
  • social.email: 메일 아이콘 링크 생성에 사용
  • social.links: SEO용 대표 외부 링크 목록
  • github.username: GitHub 아이콘 URL 자동 생성에 사용

_data/contact.yml

  • type: github 같은 기본 타입은 테마 내부 로직이 URL을 자동 생성
  • type: linkedin, type: instagram 같은 커스텀 항목은 url:을 직접 넣어야 함
  • 아이콘은 Font Awesome 클래스 기반

6. 로컬 빌드와 테스트

웹앱 실행

app/start-editor.bat

웹앱은 Node.js 내장 기능만 사용하도록 구성되어 있으며, 브라우저에서 로컬 UI를 열고 Jekyll 명령과 git 명령을 백엔드에서 실행하는 방식입니다.

기본 명령

bundle install
bundle exec jekyll b
bundle exec jekyll serve --livereload --open-url

참고 사항

  • bundle exec jekyll b 는 정적 빌드 검사용
  • bundle exec jekyll serve 는 로컬 미리보기용
  • Windows에서 detached serve는 Ruby fork 제약으로 실패할 수 있음
  • 현재 환경에서 htmlproofer는 Windows Ruby 3.4 조합에서 로컬 실패 가능성이 있음

7. GitHub Pages 배포

이 저장소는 .github/workflows/pages-deploy.yml 로 GitHub Pages를 배포합니다. 브랜치에 푸시하면 GitHub Actions가 빌드 후 배포합니다.

체크 포인트

  • GitHub 저장소 설정의 Pages source는 GitHub Actions
  • 기본 브랜치 푸시 후 Actions 성공 여부 확인
  • 배포 URL: https://jmchoi4u.github.io

8. Gemfile.lock과 플랫폼

로컬이 Windows이고 배포 환경은 Ubuntu이므로 Gemfile.lock에는 Linux 플랫폼 정보도 있어야 빌드 안정성이 높습니다. 

bundle lock --add-platform x86_64-linux
기존 .gitignore에서 Gemfile.lock을 무시하면 배포 환경 차이를 잡기 어렵습니다. 이 프로젝트에서는 lockfile을 추적하는 편이 더 안전합니다.

9. SEO와 다국어 전략

현재 가능한 것

  • 정적 HTML 사이트라서 검색엔진 크롤링 자체는 가능
  • Chirpy와 jekyll-seo-tag 기반 메타 태그 자동 생성
  • 사이트맵과 피드 생성 기반 확보

현재 한계

  • 사이트 기본 언어가 ko-KR 이므로 영어 검색 유입은 글 내용에 따라 제한적
  • 영어권 대상 확장을 제대로 하려면 같은 글의 영어 버전을 다른 URL로 제공하는 구조가 가장 좋음
  • 한국어 본문인데 제목만 영어로 바꾸는 방식은 권장되지 않음

추천 운영 방식

  • 한국어 기술 블로그를 기본으로 유지
  • 각 글에 짧은 영어 요약 섹션 추가
  • 검색 유입이 중요해진 글만 영어 버전 별도 작성
  • 대표 이미지, 설명, 제목을 글마다 다르게 작성

10. 한글 호환성과 안정성 원칙

  • 설명서와 본문은 한국어 가능, 배치 파일은 비ASCII 금지
  • 파일 이름은 되도록 영어 사용
  • CSV나 외부 내보내기는 UTF-8 기반 유지
  • 이미지 파일명, 슬러그, 폴더명은 짧고 단순하게 유지
  • 옵시디언 초안은 자유롭게 쓰되, 블로그 최종 저장은 웹앱에서 경로와 front matter를 정리하는 방식 권장

11. 웹 분석 기능 추가

웹 분석은 _config.yml 의 analytics 영역에서 켭니다. 현재 설정 파일에는 주요 서비스 자리만 미리 준비되어 있고, 값이 비어 있어서 꺼진 상태입니다.

설정 위치

analytics:
  google:
    id:
  goatcounter:
    id:
  umami:
    id:
    domain:
  matomo:
    id:
    domain:
  cloudflare:
    id:
  fathom:
    id:

추가 방법

  1. 사용할 분석 서비스를 하나 고릅니다.
  2. 해당 서비스에서 발급받은 ID를 _config.yml 에 입력합니다.
  3. bundle exec jekyll b 로 빌드 확인 후 배포합니다.
  4. 실제 사이트에서 브라우저 개발자 도구의 Network 또는 page source로 스크립트가 들어갔는지 확인합니다.

추천 운영 방식

  • 가장 단순하게 쓰려면 Cloudflare Web Analytics 또는 GoatCounter 같은 방식이 관리가 편합니다.
  • 기능이 더 많아야 하면 Google Analytics 또는 Umami를 고려할 수 있습니다.
  • 한 번에 여러 분석 스크립트를 많이 넣으면 관리가 번거로워질 수 있으니 처음에는 하나만 권장합니다.

12. 댓글 기능 추가

댓글은 _config.yml 의 comments 영역에서 켭니다. 현재는 provider가 비어 있어서 전체 댓글 기능이 꺼져 있습니다.

설정 위치

comments:
  provider:
  disqus:
    shortname:
  utterances:
    repo:
    issue_term:
  giscus:
    repo:
    repo_id:
    category:
    category_id:
    mapping:
    strict:
    input_position:
    lang:
    reactions_enabled:

어떤 댓글 방식을 고르면 좋나?

  • giscus: GitHub Discussions 기반. 현재 GitHub를 이미 쓰고 있다면 가장 자연스럽습니다.
  • utterances: GitHub Issues 기반. 비교적 단순합니다.
  • disqus: 전통적인 외부 댓글 서비스 방식입니다.

추가 순서

  1. 원하는 댓글 서비스를 먼저 외부 사이트에서 준비합니다.
  2. 리포지토리명, 카테고리 ID, shortname 같은 연결값을 발급받습니다.
  3. _config.yml 에 해당 값을 넣고 provider 를 설정합니다.
  4. 로컬 빌드 후 실제 게시글 페이지에서 댓글 영역이 보이는지 확인합니다.
현재 프로젝트는 GitHub Pages와 GitHub 기반 워크플로를 이미 사용하므로, 처음 댓글 기능을 넣을 때는 giscus 또는 utterances 가 가장 잘 맞는 편입니다.
