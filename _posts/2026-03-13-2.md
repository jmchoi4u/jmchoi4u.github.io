```md
<목표>
(여기에 목표를 적어주세요)

<중요 원칙>
- This work is important, you are being evaluated.
- 매우 중요한 작업이다. 게으르게 하지 말고 최선을 다하라.
- 사용자는 컴퓨터 초보자이므로, 항상 "처음 실행하는 사람도 바로 이해·사용 가능한가?"를 기준으로 판단하라.
- 버그가 나면 스스로 로그, 에러 메시지, 실행 흐름을 확인해 수정하라.
- 필요한 스택은 스스로 설치하라.
- 필요하면 새 작업 폴더를 만들고 시작하라.
- 루트에는 사용자에게 직접 보여야 하는 필수 파일만 두고, 나머지는 src 폴더에 정리하라.
- 가능하면 프로그램 아이콘도 제작·적용하라.
- 필요하면 실시간 웹 검색과 서브에이전트도 활용하라.
- 작업 완료 후 추가 업그레이드 아이디어를 제안하라.

<기록 및 복구>
- 향후 검토·유지보수를 위해 코드 주석은 친절하고 실용적으로 작성하라.
- 단순한 로그/버그 기록 시스템을 구축하라.
- 로그/버그 기록 시스템은:
  - 보관 기간을 제한해 용량이 과하게 커지지 않게 할 것
  - 문제 재현 시 원인 추적이 가능할 정도의 정보는 남길 것
  - 민감정보는 불필요하게 저장하지 말 것
- CHANGELOG.md에는 아래를 요약 기록하라:
  - 기록 작성 시간과 모델명
  - 사용자 요구사항
  - 변경한 파일
  - 수정 이유
  - 버그 수정 내역
  - 남은 이슈와 추후 개선점
- 삭제 기능이나 큰 수정은 되돌릴 수 있게 관리하라.

<설명서>
- 모든 설명서는 작성하고 .html 형식으로 만들며, 실행 파일처럼 눈에 띄게 노출하라.
- 제목과 내용은 모두 한국어로 작성하라.
- 설명서 종류:
  - 친절한 간단 사용 설명서
    - 컴퓨터 초보자나 어린 학생도 이해할 수 있을 정도로 쉽게 작성
    - 실행 방법, 기본 사용 흐름, 자주 하는 실수, 문제 해결 포함
  - 개발자를 위한 상세 설명서
    - 폴더 구조, 실행 방법, 주요 로직, 설정, 로그 구조, 유지보수 포인트 포함

<(MUST)품질 체크리스트>
- 완성 후 처음부터 끝까지 2회 이상 자가 검증하라.
- 검증 항목:
  □ 빌드 및 실행 성공
  □ 자주 발생하는 버그 유형 테스트
  □ 모든 버튼/메뉴 클릭 정상 동작
  □ UI 텍스트 잘림/레이아웃 깨짐 없음
  □ 앱 종료/페이지 이탈/프로세스 종료 시 관련 프로세스까지 모두 정리(cleanup)됨

<자주 있는 버그 유형들(주의)>
- 실행 스크립트, 패키징 산출물, 실행 진입점 파일 내부에는 주석 포함 한글·비ASCII 문자를 절대 넣지 마라. 인코딩/실행 오류가 날 수 있다.
- 실행파일 클릭 직후 창이 바로 꺼지는 문제를 항상 점검하라.
- 텍스트, 로그, 설정, 데이터 저장 시 한글 깨짐 가능성을 항상 고려하라.
- Excel/CSV 내보내기가 있으면 한글 인코딩 문제를 특히 주의하라.
- 화면 종료 시 관련 cmd/백그라운드 프로세스도 함께 정상 종료되게 하라.
- HTML/JS/CSS 압축·병합 시 주석 꼬임이 생기지 않게 주의하라.
- icon, image, font 등 asset 파싱 실패 가능성을 고려하라.

<UI>
- 초보자도 즉시 이해되는 명확한 구조
- 오래 봐도 피로하지 않은 저자극 화면
- 따뜻하지만 탁하지 않은 분위기
- 좁거나 조잡해 보이지 않는 여백
- 기능이 늘어나도 쉽게 무너지지 않는 확장형 레이아웃
- 한글 UI이므로 한글 가독성을 최우선으로 설계하라.
- 제목은 존재감은 주되 과장하지 말고, 프로덕트 UI처럼 절제된 계층을 유지하라.
- 본문은 오래 읽어도 편해야 하며 지나치게 작거나 빽빽하면 안 된다.
- 타이포는 화려함보다 리듬감과 위계를 중시하라.
- 선택 상태는 진한 색 블록보다 부드러운 배경 변화 + 약한 테두리/텍스트 강조를 우선하라.
- 시각적 층위(surface level)는 2~3단계를 넘겨 과하게 늘리지 마라.
- 섹션 간격, 카드 간격, 콘텐츠 최대 너비를 조정해 시각적 호흡과 그룹 구분을 분명히 하라.
- 좌측 사이드바 + 우측 메인 콘텐츠의 2단 구조
```
