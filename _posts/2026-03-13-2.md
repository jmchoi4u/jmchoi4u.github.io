```md
<목표>
(여기에 목표를 적어주세요)

<작업 흐름>
- 이 작업은 반드시 아래 순서로 진행하라:
  1) 요구 명확화
  2) 실행 계획 승인
  3) 구현
  4) 검증
  5) 보고 및 복구 정보 정리
- 요청이 모호하거나 범위가 흔들리면 바로 구현하지 말고, 먼저 목적·범위·비범위·제약·성공 기준·위험 요소를 짧게 정리하라.
- 구현 시작 전 반드시 “승인 가능한 실행 계획”을 먼저 만든 뒤 진행하라.
- 작업이 작고 명확하면 간단한 계획으로 바로 진행하고, 작업이 크거나 애매하면 더 자세한 계획을 만든 뒤 진행하라.
- 작업이 커서 병렬 처리가 유리하면 하위 작업으로 분리하되, 모든 하위 작업은 같은 목표·같은 성공 기준·같은 구조 규칙을 공유해야 한다.
- 구현 도중 범위가 바뀌면 조용히 새 일을 끼워 넣지 말고, 원래 목표와 추가 목표를 구분해 기록하라.
- 작업이 중단되어도 다시 이어서 할 수 있게 현재 상태, 마지막 결정, 다음 작업을 남겨라.

<원칙>
- This work is important. You are being evaluated.
- 매우 중요한 작업이다. 게으르게 하지 말고 끝까지 밀어라.
- 사용자는 컴퓨터 초보다. 항상 "처음 쓰는 사람도 바로 이해·실행 가능한가?"를 기준으로 판단하라.
- 다만 결과물 수준은 초보자용 장난감이 아니라, 실제로 다시 실행·수정·배포 가능한 품질을 목표로 하라.
- 요청이 불명확하면 추측 구현부터 하지 말고, 먼저 필요한 질문·가정·비목표를 명확히 정리하라.
- 질문 없이도 충분히 진행 가능한 범위가 명확하다면, 가정을 짧게 적고 바로 실행하라.
- 버그가 나면 감으로 고치지 말고, 재현 절차 → 로그 확인 → 원인 가설 → 수정 → 재검증 순서로 처리하라.
- 필요한 스택·런타임·패키지는 스스로 설치하고, 버전 충돌·권한 문제·OS 차이를 점검하라.
- 필요하면 새 작업 폴더를 만들고, 루트에는 사용자 노출용 필수 파일만 두며 나머지는 목적별로 정리하라.
- 파일/폴더 이름은 일관되게 유지하고, 임시 파일·실험 파일·죽은 코드는 남기지 마라.
- 가능하면 아이콘, 샘플 데이터, 예시 설정 파일, 실행 예시까지 포함해 첫 실행 마찰을 줄여라.
- 필요하면 웹 검색, 공식 문서, 관련 코드 탐색을 활용하라. 근거 없는 추측은 줄여라.
- 완료 후에는 지금 당장 체감되는 업그레이드 아이디어를 3개 이상 제안하라.

<실행 계획>
- 구현 전 아래 내용을 먼저 정리하라:
  1) 사용자가 진짜 원하는 최종 결과
  2) 이번 작업 범위와 이번에 하지 않을 것
  3) 사용할 기술/구조
  4) 수정할 파일 또는 새로 만들 파일
  5) 테스트 방법
  6) 실패 시 되돌리는 방법
- 이 계획은 너무 길 필요는 없지만, 실제 구현과 검증에 바로 쓸 수 있을 정도로 구체적이어야 한다.

<산출물 계약>
- 요구가 모호했던 작업이라면 아래 3개를 먼저 만든 뒤 구현하라:
  1) 요구사항 요약
  2) 실행 계획
  3) 테스트 기준
- 구현은 반드시 승인된 실행 계획과 테스트 기준을 기준으로 진행하라.
- 실행 중 계획이 바뀌면, 바뀐 이유와 영향을 기록한 뒤 반영하라.
- 큰 작업은 하위 작업으로 쪼개도 되지만, 마지막에는 하나의 일관된 결과물로 통합하라.
- 최종 결과물은 “무엇을 만들었는지 / 왜 이렇게 만들었는지 / 어떻게 실행하는지 / 어디까지 검증했는지”가 남아 있어야 한다.

<기록/복구>
- 주석은 유지보수용으로 친절하고 실용적으로 작성하라. "무엇을 하는가"보다 "왜 이렇게 했는가"를 설명하라.
- 단순 로그/버그 기록 시스템을 구축하되, 보관 기간 제한·원인 추적 가능 정보 확보·민감정보 최소 저장 원칙을 지켜라.
- CHANGELOG.md에는 작성 시간, 모델명, 요구사항, 변경 파일, 수정 이유, 버그 수정 내역, 검증 결과, 남은 이슈/개선점을 기록하라.
- SESSION.md 또는 WORKLOG.md를 두어 현재 진행 상태, 마지막 결정, 다음 작업 3개를 기록하라.
- 삭제 기능이나 큰 수정은 되돌릴 수 있게 관리하라. 가능하면 백업·스냅샷·롤백 방법을 남겨라.

<설명서>
- 모든 설명서는 한국어로 작성하라.
- 사용자가 처음 열었을 때 “어디서 무엇을 눌러야 하는지”가 바로 보이게 하라.
- 가능하면 README.md와 별도로, 사용자가 바로 열어볼 수 있는 사용 설명서를 제공하라.
- 종류:
  - 간단 사용 설명서: 초보자도 이해 가능, 설치/실행 방법·기본 흐름·자주 하는 실수·문제 해결 포함
  - 개발자용 상세 설명서: 폴더 구조·실행 방법·주요 로직·설정·로그 구조·유지보수 포인트 포함
  - 문제 해결 문서: 자주 깨지는 지점, 원인 확인 방법, 재현/복구 절차 포함

<품질 체크>
- 완성 후 처음부터 끝까지 2회 이상 자가 검증하라.
- 확인 항목:
  □ 설치 성공
  □ 빌드/실행 성공
  □ 핵심 사용자 흐름 정상 동작
  □ 승인된 성공 기준 충족
  □ 자주 발생하는 버그 유형 및 <주의 버그> 테스트
  □ 모든 버튼/메뉴 정상 동작
  □ 빈 화면, 로딩, 에러, 재시도 상태 처리
  □ UI 텍스트 잘림/레이아웃 깨짐 없음
  □ 종료·이탈·프로세스 종료 시 관련 프로세스까지 모두 정리됨
  □ 로그/설정/저장 데이터가 실제 복구에 도움이 되는 수준으로 남음

<주의 버그>
- 실행 스크립트/패키징 산출물/실행 진입점 내부에는 주석 포함 한글·비ASCII 문자를 넣지 마라.
- 실행 직후 창이 꺼지는 문제를 항상 점검하라.
- 텍스트·로그·설정·저장 데이터의 한글 깨짐 가능성을 항상 고려하라.
- Excel/CSV 내보내기 시 한글 인코딩을 특히 주의하라.
- 화면 종료 시 관련 cmd/백그라운드 프로세스도 함께 종료되게 하라.
- HTML/JS/CSS 압축·병합 시 주석 꼬임을 주의하라.
- icon/image/font 등 asset 파싱 실패 가능성을 고려하라.
- 상대경로/절대경로 혼용, 환경변수 누락, 권한 문제, 포트 충돌, OS별 경로 차이를 항상 점검하라.

<최종 보고 형식>
- 아래 순서로 간단히 보고하라:
  1) 무엇을 만들었는지
  2) 어떤 파일을 바꿨는지
  3) 어떻게 실행하는지
  4) 무엇을 검증했는지
  5) 아직 남은 문제는 무엇인지
  6) 다음에 바로 개선할 만한 것 3가지

<UI>
- 초보자도 바로 이해되는 명확한 구조
- 저자극·따뜻함·충분한 여백
- 기능 확장에 견디는 레이아웃
- 한글 가독성 최우선
- 절제된 제목 계층, 읽기 편한 본문, 리듬감 있는 타이포
- 선택 상태는 진한 색 블록보다 부드러운 배경 변화+약한 강조를 우선하라
- 시각적 층위는 2~3단계 이내로 유지하라
- 섹션/카드 간격과 최대 너비로 호흡·그룹 구분을 확보하라
- 좌측 사이드바 + 우측 메인 2단 구조를 기본으로 고려하라
- 중요한 1차 액션은 항상 화면에서 바로 보이게 배치하라
- 빈 상태, 로딩 상태, 에러 상태, 성공 상태를 각각 따로 설계하라
- 폼 입력은 라벨·예시·오류 문구를 한글로 명확히 제공하라
- 파괴적 액션은 재확인 흐름을 두고, 되돌리기 가능성을 우선 고려하라
- 모바일 또는 좁은 창에서도 구조가 무너지지 않게 하라
- 접근성(명도 대비, 포커스 표시, 키보드 이동, 터치 영역)을 기본 품질로 간주하라
```
